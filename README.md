## Вариация 2: С предварительной проверкой модели
**Идея:** Перед резервированием кредитов и постановкой в очередь выполняется быстрая валидация входных данных и проверка доступности модели. Это предотвращает бесполезное резервирование кредитов для заведомо невыполнимых задач.

```
    Клиент                FastAPI                Redis              Celery               БД
       │                     │                    │                  │                  │
       │  1. POST /predict   │                    │                  │                  │
       │────────────────────>│                    │                  │                  │
       │                     │                    │                  │                  │
       │                     │  2. Валидация      │                  │                  │
       │                     │     схемы данных   │                  │                  │
       │                     │     (Pydantic)     │                  │                  │
       │                     │                    │                  │                  │
       │                     │  3. Проверка       │                  │                  │
       │                     │     существования  │                  │                  │
       │                     │     модели         │                  │                  │
       │                     │───────────────────────────────────────────────────────>│
       │                     │                    │                  │                  │
       │                     │  4. Проверка       │                  │                  │
       │                     │     кэша модели    │                  │                  │
       │                     │     (в Redis)      │                  │                  │
       │                     │───────────────────>│                  │                  │
       │                     │                    │                  │                  │
       │  5. Если модель     │                    │                  │                  │
       │     недоступна      │                    │                  │                  │
       │     или данные      │                    │                  │                  │
       │     невалидны       │                    │                  │                  │
       │<────────────────────│                    │                  │                  │
       │  400 Bad Request    │                    │                  │                  │
       │                     │                    │                  │                  │
       │         [Если все проверки пройдены]     │                  │                  │
       │                     │                    │                  │                  │
       │                     │  6. Проверка       │                  │                  │
       │                     │     баланса        │                  │                  │
       │                     │───────────────────────────────────────────────────────>│
       │                     │                    │                  │                  │
       │                     │  7. Резервирование │                  │                  │
       │                     │     кредитов       │                  │                  │
       │                     │───────────────────────────────────────────────────────>│
       │                     │                    │                  │                  │
       │  8. Далее как в A1  │                    │                  │                  │
```

**Плюсы:**
- **Проверка наличия модели в кэше** — Redis хранит список доступных моделей и их статус (загружена/не загружена), чтобы отсечь запросы к несуществующим или выгруженным моделям до списания кредитов
- **Валидация данных** — на уровне Pydantic проверяется не только тип, но и бизнес-логика (например, диапазоны значений), чтобы не тратить кредиты на заведомо невалидные данные
- **Возврат при ошибках** — если проверки не пройдены, кредиты даже не резервируются

## Вариация 3: С приоритетными очередями

**Идея:** Используется несколько очередей в Redis с разными приоритетами. Пользователи с большим балансом или подпиской попадают в высокоприоритетную очередь, их задачи обрабатываются быстрее.

```
    Клиент                FastAPI                Redis (множество очередей)      Celery Workers
       │                     │                    │                                    │
       │  1. POST /predict   │                    │                                    │
       │────────────────────>│                    │                                    │
       │                     │                    │                                    │
       │                     │  2. Проверить      │                                    │
       │                     │     уровень        │                                    │
       │                     │     пользователя   │                                    │
       │                     │─────────────────────────────────────────────────────>│ (БД)
       │                     │                    │                                    │
       │                     │  3. Резервирование │                                    │
       │                     │     кредитов       │                                    │
       │                     │─────────────────────────────────────────────────────>│
       │                     │                    │                                    │
       │                     │  4. Выбор очереди  │                                    │
       │                     │     по приоритету: │                                    │
       │                     │     - platinum     │                                    │
       │                     │     - gold         │                                    │
       │                     │     - silver       │                                    │
       │                     │     - basic        │                                    │
       │                     │                    │                                    │
       │                     │  5. Отправить в    │                                    │
       │                     │     platinum_queue │                                    │
       │                     │───────────────────>│ (platinum)                         │
       │                     │                    │                                    │
       │                     │                    │  6. Workers забирают               │
       │                     │                    │     сначала из platinum            │
       │                     │                    │<───────────────────────────────────│ (Worker 1)
       │                     │                    │                                    │
       │                     │                    │  7. Затем из gold                  │
       │                     │                    │<───────────────────────────────────│ (Worker 2)
       │                     │                    │                                    │
       │  8. Ответ с         │                    │                                    │
       │     task_id         │                    │                                    │
       │<────────────────────│                    │                                    │
```

**Идея:**
- **Динамическое определение приоритета** — зависит от:
  - Текущего баланса пользователя (чем больше кредитов, тем выше приоритет)
  - Исторической надежности пользователя (не отправлял ли ошибочные запросы)
  В Redis создаются отдельные списки (queues): `platinum_tasks`, `gold_tasks`, `silver_tasks`, `basic_tasks`
  Celery воркеры настраиваются с разными `queue` параметрами, либо один воркер может опрашивать очереди в порядке приоритета

**Чуток креатива**
- Приоритет можно "покупать" за дополнительные кредиты. Пользователь может доплатить 2 кредита сверху, чтобы задача попала в более высокую очередь.
- Для нетерпеливых: "Ваша задача будет выполнена через 5 минут. Хотите ускорить за 3 кредита?"

```
       │                   │      возврат     │                │                 │              │

       │                   │                  │                │                 │              │
       │                   │  13. Логирование │                │                 │              │
       │                   │      инцидента   │                │                 │              │
       │                   │─────────────────────────────────────────────────────────────────>│
```

**Идея**
- **Обработка ошибок:**
  1. **На уровне воркера** — при падении задачи происходит автоматический retry (до 3 раз) с экспоненциальной задержкой
  2. **На уровне транзакции** — если после retry задача не удалась, кредиты возвращаются, статус меняется на `REFUNDED`
  3. **На уровне мониторинга** — Celery Beat ищет "потерянные" транзакции (когда задача пропала из очереди, но транзакция осталась `PENDING`) и возвращает кредиты

- **Уведомительная система:**
  - При успешном выполнении — опциональное уведомление (если пользователь подписан)
  - При ошибке с возвратом кредитов — обязательное уведомление с причиной
  - При автоматическом возврате от Beat-задачи — уведомление с пометкой "системное восстановление"

- **Аудит:** Все операции с кредитами (резервирование, списание, возврат) логируются в отдельную таблицу `credit_audit_log` для возможности разбора спорных ситуаций

## Вариация B2: С мультиплексированием потоков (Stream Multiplexing)

**Идея:** Одно WebSocket-соединение обрабатывает несколько независимых запросов одновременно. Каждый запрос получает свой `request_id`, результаты возвращаются асинхронно по мере готовности.

### Схема B2:

```
    Клиент                       FastAPI (Connection Manager)                    БД
       │                                      │                                 │
       │  1. Установка WebSocket              │                                 │
       │─────────────────────────────────────>│                                 │
       │                                      │                                 │
       │  2. Открытое соединение              │                                 │
       │<─────────────────────────────────────│                                 │
       │                                      │                                 │
       │  3. Запрос A:                        │                                 │
       │     {id:"req_1", model:"rf", ...}    │                                 │
       │─────────────────────────────────────>│                                 │
       │                                      │                                 │
       │  4. Запрос B (не дожидаясь A):       │                                 │
       │     {id:"req_2", model:"xgb", ...}   │                                 │
       │─────────────────────────────────────>│                                 │
       │                                      │                                 │
       │                                      │  5. Запуск обработки            │
       │                                      │     req_1 (фоновая задача)     │
       │                                      │                                 │
       │                                      │  6. Запуск обработки            │
       │                                      │     req_2 (фоновая задача)     │
       │                                      │                                 │
       │  7. Результат B (быстрая модель)     │                                 │
       │     {id:"req_2", result:0.8}         │                                 │
       │<─────────────────────────────────────│                                 │
       │                                      │                                 │
       │  8. Результат A (медленная модель)   │                                 │
       │     {id:"req_1", result:0.95}        │                                 │
       │<─────────────────────────────────────│                                 │
       │                                      │                                 │
       │  9. Новый запрос C                   │                                 │
       │─────────────────────────────────────>│                                 │
       │      ...                             │                                 │
```

**Идея**
- **Идентификация запросов** — каждый запрос содержит уникальный `request_id`, который возвращается в ответе
- **Параллельная обработка** — на сервере каждый запрос запускается в отдельной asyncio, результаты отправляются клиенту по мере завершения
- **Ограничение параллельности** — можно ограничить количество одновременно обрабатываемых запросов на одно соединение (например, не более 5), чтобы один клиент не исчерпал ресурсы сервера

**Преимущества перед базовым WebSocket:**
- Пользователь не ждет завершения одного запроса, чтобы отправить следующий

## Вариация B5: С подпиской на события

**Идея:** Пользователь через WebSocket подписывается на определенные события (например, "сообщить, когда модель обновится" или "уведомить о завершении batch-задачи"). Это комбинация WebSocket с паттерном Observer.

```
       │                                  │                                    │
       │  2. Подписка на события:         │                                    │
       │     {"subscribe": [               │                                    │
       │       "model:rf_v1:updates",      │                                    │
       │       "user:123:balance",         │                                    │
       │       "batch:b_456:progress"      │                                    │
       │     ]}                            │                                    │
       │─────────────────────────────────>│                                    │
       │                                  │                                    │
       │                                  │  3. Регистрация подписок          │
       │                                  │     для этого соединения          │
       │                                  │                                    │
       │  4. {"subscribed":["model..."]}  │                                    │
       │<─────────────────────────────────│                                    │
       │                                  │                                    │
       │      ... (соединение остается)   │                                    │
       │                                  │                                    │
       │                                  │  5. [Внешнее событие]              │
       │                                  │     Модель rf_v1 обновлена         │
       │                                  │                                    │
       │                                  │  6. Публикация события            │
       │                                  │     в Redis                        │
       │                                  │───────────────────────────────────>│
       │                                  │                                    │

       │                                  │     через Pub/Sub                 │
       │                                  │<───────────────────────────────────│
       │                                  │                                    │
       │  8. {"event":"model:rf_v1:      │                                    │
```
